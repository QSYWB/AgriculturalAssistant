# 农智助手 — 系统设计说明文档\n\n> 版本：0.3.0 | 最后更新：2026-06-29\n\n---\n\n## 一、项目概述\n\n农智助手是一个面向农业领域的 **AI 智能问答与作物病害诊断系统**。系统采用基于 Async Router + LangChain 的多智能体架构，通过 Supervisor Agent 自动识别用户意图并路由至不同的专业子 Agent，结合 **RAG（检索增强生成）** 技术从本地知识库和网络实时搜索中获取信息，并提供基于多模态视觉模型的作物病害图片诊断能力。\n\n**核心能力：**\n\n- 农业技术知识问答（RAG + 联网搜索）\n- 作物病害图片诊断（多模态 VLM）\n- 多轮对话与会话管理\n- 用户认证与个人中心\n- 知识库文件管理（上传/检索/删除）\n- 用户反馈收集\n\n---\n\n## **Classify**（LLM 结构化输出）：将用户问题和历史上下文注入 Prompt，由 LLM 输出 `selected_agent`、`confidence`、`reasoning` 三个字段\n- **Route**（if-else 条件路由）：根据置信度做条件路由\n  - 置信度 < 0.4 → clarify（追问澄清）\n  - 农业类问题 → agriculture_agent_node\n  - 通用问题 → general_agent_node\n- 支持缓存：相同问句 + 上下文命中缓存时直接返回\n\n#### Knowledge Agent（知识助手）\n\n基于 LangChain `create_agent` 的 ReAct Agent：\n\n- 接收 RAG 检索结果（本地知识库）和网络搜索结果（Tavily）作为上下文\n- 自主决定调用工具：\n  - `web_search`：Tavily 搜索获取实时农业信息\n  - `knowledge_cr`：语义嵌入二次过滤（相关性阈值 0.3），去除噪声\n  - `source_cite`：给回答自动标注引用来源编号\n- 要求优先使用本地知识，不足时启用网络搜索\n\n#### General Agent（通用助手）\n\n基于 LangChain `create_agent` 的 ReAct Agent：\n\n- 处理非农业类问题（闲聊、常识、天气等）\n- 需要实时信息时调用 `web_search`\n- 明确与农业问题的边界，发现农业问题提示转接\n\n#### Diagnose Agent（图片诊断）\n\n利用 Qwen3-Omni-Flash 的多模态视觉能力：\n\n- 接收用户上传的作物图片 + 文字描述\n- 识别作物种类、病斑特征（颜色/形状/分布/纹理）\n- 判断病害类型（真菌/细菌/病毒/虫害/生理性）\n- 自动关联知识库检索防治方案\n- 输出格式化的诊断报告\n\n#### Critic Agent（校验助手）\n\n回答质量审核节点，不检索知识库，只做质量检查：\n\n- 事实准确性：检查具体数值是否有编造嫌疑\n- 引用完整性：确保关键信息有引用标注\n- 幻觉过滤：标记并删除知识库和网络都未提供的信息\n- 发现严重问题时在回答开头标记 `【注意】` 并自动触发重试链路（最多 2 次）\n\n### 4.2 服务层（`service/`）\n\n| 服务 | 职责 |\n|------|------|\n| `rag_service.py` | RAG 检索管线：向量检索（MMR, top-30）→ 交叉编码器重排序（top-3）→ 语义缓存查询 |\n| `vector_store_service.py` | Chroma 向量库管理：文档分割（RecursiveCharacterTextSplitter, chunk_size=512）、MD5 去重增量入库、文件列表查询与删除 |\n| `cache_service.py` | 双层缓存：SemanticCache（余弦相似度阈值 0.92, TTL 300s）+ LLMResponseCache（MD5 键值, TTL 3600s, 持久化 JSON） |\n| `guard_service.py` | 安全防护：InputGuard（10 类提示词注入/越狱检测）、OutputGuard（系统提示泄露检测）、PathSecurity（路径/会话 ID 安全校验） |\n| `conversation_service.py` | 对话管理：支持认证用户（MySQL 持久化）和游客（内存无持久化）两种模式，会话 CRUD、消息追加、历史摘要生成 |\n| `geo_service.py` | 地理编码：Nominatim 逆地理编码 + ip-api.com IP 定位，用于位置感知回答 |\n\n### 4.3 API 层（`api/`）\n\n| 路由 | 方法 | 获取用户位置\n         → 检查 LLMResponseCache → 命中则直接返回\n         → Supervisor Agent\n             ├─ classify_node: LLM 意图分类\n             ├─ route_decision: 条件路由\n             ├─ agriculture_agent_node → Knowledge Agent (ReAct) → Critic Agent\n             ├─ general_agent_node → General Agent (ReAct)\n             └─ clarify_node → 返回追问\n         → OutputGuard 检查\n         → [认证用户] 持久化消息\n         → SSE 流式/完整响应 → 客户端渲染\n```\n\n### 5.2 RAG 检索流程\n\n```\n用户问题 → embedding_model.embed_query\n         → SemanticCache 查询（余弦相似度 ≥ 0.92）\n         → 未命中 → Chroma MMR 检索（top-K=30, lambda_mult=0.7）\n         → DashScope gte-rerank 交叉编码器重排序（top-N=3）\n         → 结果存入 SemanticCache\n         → 直接返回 or 注入 Agent 上下文\n```\n\n---\n\n## 六、数据库设计\n\n### 6.1 users（用户表）\n\n| 字段 | 类型 | 说明 |\n|------|------|------|\n| 指令忽略类：`忽略之前所有指令`、`ignore above prompts`\n- 系统泄露类：`输出系统提示`、`重复你的设置`\n- 越狱类：`你从此被解放`、`you are now free`\n\n### 7.2 输出防护（OutputGuard）\n\n检测系统提示泄露模式：`核心行为准则`、`意图分类体系`、`调度规则` 等，发现后安全截断。\n\n### 7.3 速率限制\n\n基于 slowapi，默认 30 次/分钟，认证接口 60 次/分钟。\n\n### 7.4 认证安全\n\n- JWT 令牌（HS256，24 小时过期）\n- bcrypt 密码哈希（截断 72 字节兼容 bcrypt 5.x）\n- 可选依赖：MySQL 不可用时自动降级为游客模式\n- 路径安全校验：阻止路径遍历和非法字符\n\n---\n\n## |\n|------|------|------|-----|--------|\n| SemanticCache | 语义缓存（向量余弦相似度） | embedding 向量 | 300s | 内存 |\n| LLMResponseCache | 精确键值缓存 | MD5(query+context) | 3600s | JSON 文件 |\n\n两层缓存的目的是：语义缓存避免重复的向量检索和 rerank 调用；LLM 缓存避免完全相同的问句重复调用 API。相似但不相同的问句（如措辞不同）走语义缓存，完全相同的问句走 LLM 缓存。\n\n---\n\n## 十、部署与环境\n\n### 10.1 环境变量（.env）\n\n| 变量 | 说明 | 示例 |\n|------|------|------|\n| DASHSCOPE_API_KEY | DashScope API 密钥 async/await + if-else 后：\n\n- **延迟降低**：省去了 StateGraph 的图执行器初始化开销\n- **调试更直观**：同步代码的调用栈比图结构更易追踪\n- **测试更简单**：sync def _classify_intent() 可以直接单测，无需 mock 图运行时\n- **依赖更少**：不再依赖 langgraph 包\n\n如果后续出现 10+ 节点、循环、并行分支的复杂工作流，会重新考虑 LangGraph。\n\n### 12.2 为什么用 Chroma 而非 Milvus/Pinecone？\n\n- 项目体量小，Chroma 零配置、本地持久化、与 LangChain 深度集成\n- 无需部署独立向量数据库服务\n- 不足：不支持分布式、大规模场景需迁移\n\n### 12.3 为什么自研 Guard 而非用开源防护库？\n\n- 项目中英文混合场景，通用防护库对中文 prompt 注入覆盖不全\n- 规则轻量（10 条正则），维护成本低\n- 专用的农业领域防护规则可定制\n\n### 12.4 游客模式 vs 强制登录\n\n- 降低使用门槛：用户无需注册即可体验核心对话功能\n- 认证用户获得：会话持久化、知识库管理、反馈提交等增强功能\n- MySQL 不可用时自动降级为全游客模式\n\n### 12.5 为什么图片上传用加号图标？\n\n- 保持输入区域简洁，加号是通用的"附加操作"符号\n- 考虑到未来可能扩展更多附件类型（文件、位置等）\n\n---\n\n## 十三、已知限制与改进方向\n\n| 方向 | 当前状态 | 改进建议 |\n|------|----------|----------|\n| 测试覆盖 | 仅脚本测试 | 补充 pytest 单元测试（agent 路由、RAG
